# ai-dev-council 要件定義

Version: 1.0.0

---

# 1. 目的・背景

タスク説明（自由記述）から、実際に動くコード＋テストを自律生成し、その過程
（設計・設計レビュー・実装・実装レビュー）をGitHub issueとして記録する。

姉妹ツール[ai-council](https://github.com/momokuomomo-crypto/ai-council)は
「テーマについて話し合う」会話専用ツール（実際のコードは書かない）だが、
ai-dev-councilは実際にファイルシステムを操作し、bashコマンド（テスト実行等）
を実行する、プログラミング特化版である。

---

# 2. スコープ

## 2.1 ai-dev-council自体の機能として扱う範囲

* タスク説明からの設計生成（OpenAI）
* 設計レビュー（Claude+Gemini、リビジョンループ）
* 実装（Claude Agent SDKによる自律コーディング、output_dir配下へのファイル
  作成・bash実行）
* 実装レビュー（Gemini+OpenAI、リビジョンループ、指摘があれば自律修正）
* テストの実際の実行と結果記録（`pytest`、生ログ＋結果CSV）
* 実行記録のGitHub issue化（ai-dev-council自身のリポジトリに、closeせず
  残す）
* 実行前の2段階の`[Y/n]`確認（固定回数API呼び出し／自律エージェント実行）
* 1日あたりの実行回数制限

## 2.2 ai-dev-council自体の機能として扱わない範囲

* 要件・仕様の合意形成そのもの（ai-councilの役割。3社による独立意見→
  議論→統合はai-council側で行い、ai-dev-councilはその結果を入力として
  受け取るのみ）
* 生成したコードの本番デプロイ・公開作業
* ポートフォリオ成形・対外提示用フォーマットへの変換等、個別案件固有の
  後工程（各案件の要件定義・設計書側のスコープ）

# 3. 用語定義

| 用語 | 意味 |
|---|---|
| タスク説明 | 作りたいソフトウェアの自由記述の説明（CLIの必須引数） |
| 設計ドキュメント | OpenAIが生成する構造化された設計（overview/requirements/architecture/file_plan/test_plan/open_questions） |
| 設計レビュー | Claude+Geminiによる設計ドキュメントの承認/指摘判定 |
| 実装 | Claude Agent SDKによる自律コーディング（output_dir配下へのファイル作成・bash実行） |
| 実装レビュー | Gemini+OpenAIによる生成コードの承認/指摘判定 |
| ai-council | 要件定義・設計の合意形成を担う姉妹ツール（[こちらの要件定義参照](https://github.com/momokuomomo-crypto/ai-council/blob/master/docs/要件定義.md)） |

# 4. 機能要件

## 4.1 入力

* `task`（必須）：作りたいソフトウェアの説明（自由記述）。
* `--output-dir`（必須）：コードの生成先ディレクトリ。
* `--context-file`（任意、現状）：参考情報を含むテキストファイル。
  ai-councilが出力した`summary.txt`等を渡すことを想定。
* `--max-rounds`（任意、デフォルト1）：設計レビューのリビジョンループ上限回数。
* `--max-implementation-rounds`（任意、デフォルト1）：実装レビュー→修正の
  リビジョンループ上限回数。

## 4.2 パイプラインステージ

1. **設計**（OpenAI）：タスク説明＋contextから設計ドキュメントを生成する。
2. **設計レビュー**（Claude+Gemini、最大`--max-rounds`回）：各社が独立に
   承認/指摘を返す。非承認ならOpenAIが改訂し、次ラウンドへ。全員承認で
   早期終了。
3. **実装**（Claude Agent SDK）：確定した設計ドキュメントをもとに、
   `--output-dir`配下へファイルを作成・編集し、bashコマンド（テスト実行等）
   を自律的に実行する。
4. **実装レビュー**（Gemini+OpenAI、最大`--max-implementation-rounds`回）：
   生成コードを独立にレビューする。非承認ならClaude Agent SDKが修正し、
   次ラウンドへ。全員承認で早期終了。
5. **記録**：全ステージの結果をGitHub issue（ai-dev-council自身のリポジトリ）
   として作成する（closeしない）。

各プロバイダーは自分が書いた成果物をレビューしない（設計＝OpenAI→レビューは
Claude+Gemini、実装＝Claude→レビューはGemini+OpenAI）。

## 4.3 安全ガード

* `confirm_api_calls`：固定回数API呼び出し（設計・設計レビュー・実装レビュー）
  の見込み回数を実行前に表示し、`[Y/n]`確認する。
* `confirm_agent_run`：Claude Agent SDKによる自律実行（実装・実装修正）の
  直前に、対象ディレクトリと最大ターン数を明示して別途確認する（多ターン・
  変動費用であるため、固定回数API呼び出しとは別ゲート）。
* `usage_tracker`：両方の確認を通過した後、パイプライン開始前に1日あたりの
  実行回数を1回だけカウントする（どちらかの確認で断れば消費しない）。
* `permission_mode`はデフォルト`acceptEdits`（`bypassPermissions`は使わず、
  許可外の操作はフェイルセーフに倒す）。

## 4.4 テスト実行記録

* エージェントの自己申告に加え、実装直後と実装レビュー完了後の2時点で、
  パイプライン側が独立して実際に`pytest`を実行し、結果を
  `<output-dir>/test_logs/`配下に保存する。
  * `{タイムスタンプ}_{implementation|final}_pytest_log.txt`（生ログ全文）
  * `{タイムスタンプ}_{implementation|final}_test_results.csv`（テスト
    ケース単位の結果一覧）
* GitHub issueに最終テスト結果（成功/失敗件数とファイルパス）を記載する。

## 4.5 GitHub issueによる記録

* 1回の実行につき1つのissueを、ai-dev-council自身のリポジトリに作成する。
* ai-councilの開発ワークフロー用issue（実装コミット参照でclose）とは異なり、
  「1回の実行の記録・成果物」であるため、作成後もOPENのまま残す。

## 4.6 【新規要件・未実装】実装の前提条件：要件定義書・設計書の存在確認

**現状（4.1〜4.5）は、`--context-file`が任意であり、タスク説明だけでも
実装（output_dirへのコード生成）まで進んでしまう。** これは、ai-councilで
要件定義・設計を固めて人間が確認してからai-dev-councilに進む、という
[運用フロー](https://github.com/momokuomomo-crypto/ai-council/blob/master/docs/運用フロー.md)
を、ai-dev-council側では強制していないことを意味する。

これを是正するため、以下を新しい要件とする（**本ドキュメント作成時点では
未実装。設計・実装は次フェーズで行う**）。

* 実装ステージ（設計以降のすべてのステージ）は、対象案件の**要件定義書・
  設計書が既に存在する場合にのみ**実行する。存在しない場合はエラーで
  終了し、API呼び出し・自律コーディングエージェントを一切起動しない。
* 具体的には、`--context-file`（任意入力）を**必須**に変更するか、または
  `--requirements-file`・`--design-file`のような専用引数を新設し、
  ファイルが存在しない・指定されていない場合は起動時点でエラーとする案が
  考えられる（詳細は設計フェーズで確定する）。
* これらのファイルは通常、ai-council側で作成され`output/{slug}/`に置かれる
  `要件定義_*.md`・`設計書_*.md`（人間のレビューを経たもの）を想定する。
  ただし、ファイルの中身が実際に人間のレビューを経たものかどうかまでを
  プログラム的に検証することはできない。「要件定義書・設計書を用意してから
  実装する」という運用ルールの徹底は、引き続き人間の運用（および
  かんたんセットアップガイドでの案内）に委ねる。
* ai-councilとai-dev-councilは別リポジトリ・別ローカルパスにクローンされて
  いる場合があるため、ファイルパスの受け渡しは相対パスの自動解決に頼らず、
  明示的な引数指定を必須とする。

# 5. 非機能要件

## 5.1 セキュリティ

* 各プロバイダーのAPIキーは`.env`（git管理対象外）で管理する。
* Claude Agent SDKの操作範囲は`--output-dir`にスコープを限定し、
  `permission_mode: acceptEdits`でフェイルセーフに倒す。

## 5.2 コスト管理

* 4.3の2段階`[Y/n]`確認と`max_runs_per_day`（デフォルト3）により、意図しない
  課金・自律実行を防ぐ。

## 5.3 堅牢性

* Claude Agent SDKの`result_message.subtype`が`"success"`以外
  （`error_max_turns`等）でもクラッシュさせず、「実装が完了しなかった」
  という結果として扱い、issueにその旨を明記する。
* 修正ループはセッション再開APIに依存せず、毎回新規`query()`呼び出しで
  「既存コードはoutput_dir内にあるので自分でReadせよ」という形にすることで、
  SDKのバージョン差異に対して頑健にする。

---

# 6. 制約条件

* 実装ステージはClaude（Agent SDK）固定、設計はOpenAI固定、レビュー担当は
  「自分が書いた成果物はレビューしない」組み合わせで固定する。将来変更する
  場合は[設計.md](設計.md)を更新する。
* 実装は`--output-dir`にスコープを限定し、それ以外のファイルシステム操作は
  想定しない。

---

# 7. ai-councilとの関係・運用フロー

* ai-council（要件定義・設計の合意形成、会話専用）→人間による確認・承認
  （必須）→ai-dev-council（実装）の2フェーズ構成とする。連続実行はしない。
* 現状はこの運用ルールがドキュメント上の約束事にとどまっており、
  ai-dev-council側のコードでは強制されていない（4.6）。
* 詳細は[ai-councilのdocs/運用フロー.md](https://github.com/momokuomomo-crypto/ai-council/blob/master/docs/運用フロー.md)を参照。
