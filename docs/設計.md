# ai-dev-council 設計書

## 1. 目的

タスク説明（自由記述）から、実際に動くコード＋テストを自律生成し、
その過程（設計・レビュー・実装・レビュー）をGitHub issueとして記録する。

ai-council（会話専用、意見→議論→統合）とは異なり、本ツールは実際に
ファイルシステムを操作し、bashコマンド（テスト実行等）を実行する。

## 2. ステージ構成

| ステージ | 担当 | 入力 | 出力 |
|---|---|---|---|
| 設計 | OpenAI | タスク説明 | 設計ドキュメント |
| 設計レビュー | Claude + Gemini | 設計ドキュメント | レビューフィードバック（承認/指摘） |
| （不承認時）設計改訂 | OpenAI | 設計ドキュメント＋フィードバック | 改訂された設計ドキュメント |
| 実装 | Claude（Agent SDK） | 設計ドキュメント | output_dir配下のコード＋テスト |
| 実装レビュー | Gemini + OpenAI | 設計ドキュメント＋生成コード | レビューフィードバック（承認/指摘） |
| （不承認時）実装修正 | Claude（Agent SDK） | フィードバック | 修正されたコード |
| 記録 | - | 全ステージの結果 | GitHub issue（このリポジトリ、OPENのまま） |

各プロバイダーは自分が書いた成果物をレビューしない（設計＝OpenAI→レビュー
はClaude+Gemini、実装＝Claude→レビューはGemini+OpenAI）。

## 3. スキーマ

### 3.1 設計ドキュメント

```
overview: string              # 概要
requirements: string[]        # 機能要件・非機能要件
architecture: string           # 技術選定・全体構成
file_plan: {path, purpose}[]   # 作成予定ファイルと役割
test_plan: string               # テスト方針
open_questions: string[]        # 未決定・要確認の論点
```

### 3.2 レビューフィードバック（設計レビュー・実装レビュー共通）

```
approved: boolean
issues: {severity: "blocker"|"major"|"minor", location: string, description: string}[]
suggestions: string[]
```

## 4. 実装ステージの特殊性

`claude_coding_agent.py`はClaude Agent SDK（`claude_agent_sdk`）を用いる。
他のプロバイダー呼び出し（`*_provider.py`）がスキーマ強制の単発API呼び出しで
あるのに対し、実装ステージは以下の点で根本的に異なる：

- `cwd`（output_dir）にスコープを限定した、ファイル書き込み・bash実行を
  伴う自律的・多ターンの実行である。
- 固定回数のAPI呼び出しではなく、実行するまで所要ターン数・費用が
  確定しない。
- 実行前に専用の警告付き`[Y/n]`確認（`confirm_agent_run`）を要求する
  （通常の`confirm_api_calls`とは別ゲート）。

修正ループ（`run_implementation_fix`）はセッション再開APIに依存せず、
毎回新規`query()`呼び出しで「既存コードはoutput_dir内にあるので自分で
Readせよ、レビュー指摘はこれ」という形にする。これによりSDKのバージョン
差異（セッション管理APIの変化）に対して頑健になる。

`result_message.subtype`が`"success"`以外（`error_max_turns`等）でも
クラッシュさせず、「実装が完了しなかった」という結果として扱い、
issueにその旨を明記する。

## 5. 安全ガード

1. `confirm_api_calls`：固定回数API呼び出し（設計・設計レビュー・実装
   レビュー）の見込み回数を表示して確認する。
2. `confirm_agent_run`：Claude Agent SDKによる自律実行の直前に、対象
   ディレクトリと最大ターン数を明示して確認する。
3. `usage_tracker`：1日あたりの実行回数上限（両方の確認を通過した後、
   パイプライン開始前に1回だけカウントする。どちらかで断れば消費しない）。

`permission_mode`はデフォルト`acceptEdits`（`bypassPermissions`は使わず、
許可外の操作はフェイルセーフに倒す）。

## 6. GitHub issue

`gh issue create`で、このリポジトリ（ai-dev-council自身）にissueを作成する。
ai-councilの開発ワークフロー用issue（実装コミット参照でclose）とは異なり、
これは「1回の実行の記録・成果物」であるため、作成後もOPENのまま残す。
